Domain

This section explains the financial domain the system operates in. You do not need to be a trader to understand this — every concept is explained from first principles.

1.1 What is Nifty 50 and NSE?

NSE (National Stock Exchange of India) is India's largest stock exchange by trading volume. It operates from 9:15 AM to 3:30 PM, Monday to Friday (excluding market holidays).

Nifty 50 is NSE's flagship stock market index. It tracks the performance of 50 of the largest and most liquid companies listed on NSE — companies like Reliance, HDFC Bank, TCS, and Infosys. When traders or news say "the market is up 1%", they usually mean the Nifty 50 moved up 1%.

The current value of Nifty 50 is called the spot price (e.g., 24,013 points). This is what ZuppTrade monitors continuously.

1.2 Options Trading — Core Concepts

ZuppTrade trades options on the Nifty 50 index. An option is a financial contract that gives the buyer the right (but not the obligation) to buy or sell Nifty at a specific price on a specific date.

1.3 Trading Strategies Used

ZuppTrade does not take naked (unlimited risk) positions. It always trades spreads — simultaneously buying and selling options to limit maximum loss.

StrategyWhen UsedHow it WorksRisk Profile
Bull Put Spread Market is bullish or mildly bullish SELL a put at strike K1, BUY a put at lower strike K2. Collect net premium upfront. Profit if Nifty stays above K1. Max profit = net premium collected. Max loss = spread width minus net premium.
Bear Call Spread Market is bearish SELL a call at strike K1, BUY a call at higher strike K2. Collect net premium. Profit if Nifty stays below K1. Max profit = net premium collected. Max loss = spread width minus net premium.
Bull Call Spread Market is strongly bullish (Extreme) BUY a call at K1, SELL a call at higher K2. Pay net premium. Profit if Nifty rises above K1. Max loss = premium paid. Max profit = spread width minus net premium.
Iron Condor Market is neutral (sideways expected) Combines a Bull Put Spread and Bear Call Spread. Profit if Nifty stays within a range. Limited profit, limited loss. Works best in low-volatility range-bound markets.
SKIP Market conditions are unclear or VIX is extreme (>24) No trade placed. System alerts user. No risk, no reward. Safety first.

Functional Design

2.1 What the System Does and Why

ZuppTrade is an automated decision-support and execution system for trading Nifty 50 weekly options. It replaces hours of manual market analysis with a disciplined, algorithm-driven workflow.

Core Philosophy Consistency over return maximisation. Every trade decision must be explainable from its inputs. No gut feel. No overrides without a reason. The system aims to produce the same trade recommendation given the same market data — every single time.

What it does:

  • Collects market data from multiple sources: Upstox (option chain, live prices), NSE India (FII/DII flows), Marketaux (news sentiment), and AI-assisted commentary extraction
  • Analyses data through a 5-tier scoring model to determine the market direction (Bullish/Bearish/Neutral)
  • If conditions are right, selects a strategy, finds the optimal strike prices, validates risk gates, and sizes the position
  • Presents a trade card to the user for confirmation
  • On confirmation, executes both legs of the spread on Upstox simultaneously
  • Monitors the live trade every 5 minutes and alerts or auto-exits based on predefined thresholds

What it does NOT do: It does not trade without user confirmation (semi-automatic). It does not place trades when VIX is extreme (>24). It does not chase high returns at the cost of high risk.

User opens the ZuppTrade UI (Angular web app). Reviews the market signal — Bias (Bullish/Bearish/Neutral), Strength (Extreme/Mild/Weak), Confidence score, and breakdown by tier. If satisfied, clicks "Get Recommendation".

Agent 2 runs the 5-layer algorithm: selects strategy → calculates expected move → selects strike prices → validates 4 gates → sizes lots. Presents a trade card valid for 20 minutes.

User reviews trade card: strategy, strikes, expected premium, max profit, max loss, PoP, lot count. Can optionally change the number of lots. Clicks CONFIRM to execute or REJECT to skip.

9:33 AM

Agent 5 checks available margin, places both legs simultaneously via Upstox multi-leg order API. Polls for fills (30s timeout). If both legs fill — trade is ACTIVE. If one leg fails, the other is rolled back.

Every 5 min

Checks live Nifty spot against T1/T2/T3 thresholds. Updates live P&L. Sends alerts on WATCH/READJUST. Automatically exits via Agent 5 if T3 (exit threshold) is breached.

If the trade was not exited earlier, options expire. OTM options expire worthless — the net premium collected is the profit. ITM options are settled by NSE automatically.

2.3 The Five Agents

Agent 1 — Market Analyst
Port 8081

The "brain" that reads the market. Collects data from 5 independent sources, scores each signal +1/0/-1, weights them, and produces a composite market direction signal.

  • 5-tier scoring: Price Structure, Technical, Institutional, Volatility, Sentiment
  • Outputs: Bias, Strength, Confidence, composite score (−1 to +1)
  • Writes signal to database; Agent 2 reads from there
Agent 2 — Recommender
Port 8082

The "strategist". Takes Agent 1's signal and live market data, runs a 5-layer algorithm to produce an actionable trade card with specific strikes, lot count, and risk/reward.

  • Selects strategy based on bias + VIX + IV regime
  • Calculates expected move using Black-Scholes + straddle price
  • Validates 4 hard gates (PoP, max loss, PoPP gap, RoC)
  • Sizes lots based on capital and risk limits
Agent 3 — Monitor
Port 8083

The "watchdog". Runs every 5 minutes during market hours. Checks live Nifty against pre-set exit thresholds. Orchestrates readjustments and automated exits.

  • T1 Watch → recalculate live PoP
  • T2 Readjust → roll to a safer strike
  • T3 Exit → close all legs immediately
  • Pauses automatically when VIX > 24
Agent 5 — Executor
Port 8085

The "hands". Interfaces directly with Upstox order APIs. Places, monitors, modifies, and cancels orders. Handles fill confirmation and slippage detection.

  • Margin check before any order
  • Simultaneous multi-leg entry via Upstox multi/place
  • 30-second fill polling with LIMIT→MARKET fallback
  • Rollback if one leg fails
Supporting Modules
Shared Libraries
  • core-module — Upstox REST client, AlertService, ExpiryDateService. Shared as a Maven library by all agents.
  • shared-domain — All enums, DTOs, constants, custom exceptions. The "common language" of the system.
  • ledger-module — Event-sourcing audit trail. Every trade state change is immutably recorded.
  • db-migrations — Single source of Flyway SQL migrations. All agents share the same schema.

2.4 Key Business Rules

Each tier collects signals, scores each +1 (bullish), 0 (neutral), or −1 (bearish), and computes an average. Missing data scores 0 — the system never throws an error for missing data.

IV Regime: IV/HV ratio >1.20 = Rich, <0.85 = Cheap, else Fair.

Layer 2: Expected Move Calculation

Two methods are computed and cross-checked:

// Method 1 — Straddle price (market consensus)
market_expected_move = ATM_CE_LTP + ATM_PE_LTP

// Method 2 — Black-Scholes (mathematical)
EM = spot × IV × √(DTE / 365)
one_four_sd_boundary = spot ± (EM × 1.4)   // 84% probability zone

// If divergence between methods > 15%: flag, use more conservative boundary

Layer 3: Strike Selection Rules

  • Short strike: at or beyond the 1.4 SD boundary, delta ≤ 0.20, rounded to nearest 50
  • Spread width: 50–150 points for ₹5L capital; proportionally wider for larger capital
  • Long strike: short_strike ± spread_width (direction depends on strategy)
  • Verify |PoP − PoPP| ≤ 15% — if not, try next strike

Layer 4: Gate Validation (Chain of Responsibility Pattern)

Gate 1
PoP — Probability of Profit
Black-Scholes seller's probability ≥ 80%. Computed as N(d2) from the short strike's IV.
HARD REJECT — no trade if fails
Gate 2
Max Loss vs Capital
Real expected loss (50% of max theoretical loss) ≤ 1.5% of capital. Used for lot sizing, not rejection.
INDICATIVE — sizes lots, never rejects
Gate 3
PoP–PoPP Gap
|PoP − PoPP| ≤ 15%. Prevents selecting strikes where profit probability degrades sharply inside the spread.
HARD REJECT — no trade if fails
Gate 4
Minimum Return on Capital
RoC ≥ 0.5% × (DTE ÷ 5). Adjusted for time to expiry — short DTE trades need proportionally less premium.
HARD REJECT — no trade if fails

Black-Scholes PoP Formula

// d1, d2 from Black-Scholes model:
d1 = (ln(S/K) + (r + 0.5σ²) × t) / (σ × √t)
d2 = d1 − σ × √t

// PoP for SHORT PUT (seller's probability = put expires OTM):
PoP = N(d2)     // N = standard normal CDF, Apache Commons Math

// Where:
// S = Nifty spot price
// K = strike price of the short put
// r = 0.065 (India repo rate, risk-free rate)
// σ = IV of the strike (from Upstox option chain, divided by 100)
// t = DTE / 365
// All inputs/outputs BigDecimal. Convert to double only for Math.log/sqrt calls.

Layer 5: Position Sizing

real_expected_loss_per_lot = max_loss_per_lot × 0.50
// (Agent 3 exits at T2 = 50% of max loss, so we size to that, not theoretical max)

max_loss_allowed = capital × 0.015      // 1.5% max loss rule
lots_from_loss   = floor(max_loss_allowed / real_expected_loss_per_lot)
lots_from_margin = floor(capital / SPAN_margin_per_lot)

final_lots = min(lots_from_loss, lots_from_margin)

4.4 Agent 3 — Monitoring Engine (Low-Level)

Agent 3 runs every 5 minutes between 9:15 AM and 3:30 PM using Spring @Scheduled with ShedLock to prevent overlapping runs.

Monitoring Thresholds for Credit Spreads (Bull Put / Bear Call)

T1 — Watch
Nifty reaches short_strike + 150 pts
Recalculate live PoP.
HOLD if PoP ≥ 75%
T2 — Readjust
Nifty reaches short_strike + 75 pts
OR MTM loss = 50% of max loss
Roll short strike lower
OR tighten spread
T3 — Exit
Nifty breaches short_strike
OR MTM loss = real_expected_loss
Close ALL legs — market order
Agent 5 executes immediately

Dynamic Overrides

ConditionOverride
VIX spikes > 30% intradayForce T3 EXIT regardless of Nifty price
VIX > 24 (Extreme)PAUSE monitoring — alert user to decide manually

Readjustment Flow (T2 Action)

When T2 is triggered and the system decides to readjust rather than exit:

  1. Check DTE ≥ 2 days (if DTE < 2, exit instead — too close to expiry to readjust)
  2. Call Agent 5 to exit the current trade (reverse market order)
  3. Fetch fresh Agent 1 signal (on-demand call)
  4. Calculate relaxed Gate 1 threshold: 65% if VIX ≤ 22, 70% if VIX > 22
  5. Call Agent 2 /recommend with the relaxed PoP gate
  6. Auto-confirm and call Agent 5 to execute the new trade

4.5 Agent 5 — Execution Engine (Low-Level)

Order Execution Sequence

  1. Receive ExecuteTradeRequest (trade ID, legs with instrument key, action, limit price, quantity)
  2. Verify trade status is CONFIRMED in database
  3. Margin check: POST /v2/charges/margin on production Upstox API. Reject if insufficient.
  4. Place both legs simultaneously: POST /v2/order/multi/place on HFT Upstox endpoint
  5. Poll both order statuses every 5 seconds for up to 30 seconds
  6. If still open after 30 seconds: modify order from LIMIT → MARKET (or cancel in sandbox mode)
  7. If one leg fails after the other fills: place reverse MARKET order on the filled leg (rollback). Alert user.
  8. Both fills confirmed → compute actual net premium
  9. Slippage check: actual premium < expected × 90% → set slippage alert flag (trade still proceeds)
  10. Write fill data to trades.entry_fills, update status to ACTIVE

Agent 1 — Market Analyst (:8081)

MethodPathDescription

Authentication

v1 — API Key All inter-agent calls use the X-API-Key header. The key is stored in the environment variable X_API_KEY on each agent. OAuth2 for end-user login is planned for v2 and the security layer is designed to be swappable without touching business logic.
